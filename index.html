The one-liner

One conductor, thirteen specialists

pre-iPRD doesn’t audit the design itself — it conducts thirteen specialist agents through a fixed sequence. A pasted Figma link and the words “pre iprd” are all it needs to begin. After you confirm the platform once, it runs end to end on its own: nothing skipped, no frame deleted, no endpoint or screen invented, and every change it makes read straight back to prove it landed. The whole run answers to a single master design system — and nothing else.

Division of labour

Analysts plan. The main session acts.

Only the main session can write to Figma through the use_figma plugin — a write attempted from a spawned agent turns into plain text and stalls the run. So the work splits cleanly in two:

⌕The analysts — every agent

Spawned with the file key, node id and whatever earlier steps produced, each agent reads the design and hands back findings plus a fix-plan. It never touches the file.

✎The hands — main session

It applies every write the plan calls for, then reads it straight back to confirm it stuck. If an agent can’t even read Figma, the main session does that agent’s job itself.

The assembly line · 9 steps

Read → plan → apply → verify

With the rulebook chosen, nine build-and-fix agents run back to back. Each reads, hands over a plan, and the main session applies it and checks the result before the next begins.

1

Sets the file up file-creator-agent

Preps the file for work: renames the design page to “Ready for engineering”, adds a “Design playground” page with a branded thumbnail, and imports the design system’s variables so the agents that follow can bind real tokens. (Renaming the file itself stays a manual step.)

2

Names every frame file-renamer-agent

Gives each frame a unique, canonical name — platform / area / feature / module / screen — state — so nothing is ambiguous downstream. An “Untitled” frame is a hard blocker.

3

Sorts into sections section-splitter

Gathers loose screens into one named section per sub-flow, top-aligns and pads them on a shared baseline, and proves nothing spills outside its section.

4

Binds the tokens token-matcher

The heavy lifter. Snaps every off-grid value to the nearest token (±4px) and binds colours, text, shadows and spacing to real variables in the right platform mode — while leaving component instances, logos and illustrations untouched.

5

Fixes the data data-matcher

Swaps out placeholder copy and corrects what’s wrong — dates, amounts, percentages, plus/minus signs, totals that don’t add up, and rows repeating the same value. Anything needing a real business rule is flagged, never guessed.

6

Builds the missing states edge-finder

Finds the states a flow forgot — loading, empty, error, success — and builds them by cloning the nearest screen from real components into a “🆕 Proposed” section. ⛔ Then it pauses for the designer to keep or drop them.

7

Specs the motion animation-interaction-suggester

Writes one reference frame covering transitions, micro-interactions and timing (page 300ms · modal 250ms · skeleton 1.5s) — guidance for engineering, never a change to the live screens.

8

Annotates for handoff annotater-agent

Marks up every key frame with where it goes, what it calls, what each control does, and what QA should test. Unknown endpoints are labelled TBD, never invented.

9

Maps the navigation redirection-helper

Builds one reference frame listing every touch-point — buttons, links, filters, actions, back — and where each leads, calling out any dead-end CTA.

Quality gate

The reviewer — loop until it’s all green

A read-only reviewer now checks every frame against every dimension at once — naming, layout, tokens, data, edge states, annotations, navigation — and returns a per-frame pass/fail scorecard.

✓Everything passes

Clear to build the flow chart.

↻Something fails

The reviewer points to the exact agent and frames at fault. Those re-run, the writes re-apply, and the review runs again — looping until the scorecard is fully green. The flow chart can’t start before then.

Step 12

The flow chart — once it’s green

The user-flow-creator returns a node-and-edge model and a build spec; the main session draws the chart through Figma. The output is a clean, interaction-level map — no crossing lines, API badges on the right edges, spacing bound to design-system variables — and it’s checked so no line sits on a tile, no tiles overlap, and every screen is present.

Step 13

Self-check & report

Last, the self-check-reporter runs a final sweep — names, spacing, token conformance, a clean chart — and writes up what it found, what it fixed, and what still needs a human. It posts the summary to Slack #pre-iprd-test with deep-links to every deliverable, owners named by role, text only.

Always true

The standing rules
  • Agents read and plan; only the main session writes — and it reads every change back to be sure it landed.
  • The flow chart waits for a fully green review.
  • After Step 0, no more questions — and no step is ever skipped.
  • Frames are never deleted, only hidden; a section is emptied before it’s removed.
  • Nothing is invented — unknowns are marked TBD, and missing screens are cloned, not drawn.
  • One master design system per run, with just three places it’s ever allowed to pause.
