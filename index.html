What it is

An orchestrator, not a lone auditor

The pre-iPRD Design Audit Agent does not do the audit work alone. It drives a pipeline of twelve specialised sub-agents in a fixed order. It is triggered whenever a designer pastes a Figma page or section link and asks for a “pre iprd” review in any phrasing — the pasted link by itself is a valid invocation. Once a single platform-confirmation gate is cleared, the agent runs the whole pipeline autonomously: no further questions, nothing skipped, no frame ever deleted, no API endpoint or screen ever invented, and every change written into Figma verified by reading it back. Throughout, it conforms strictly to one confirmed master design system and nothing else.

How it operates

Analyst + main-hands

The Figma write channel — the use_figma plugin — is reachable only from the main orchestrating session. A write attempted from a spawned sub-agent serialises into plain text and halts the run. So the pipeline splits into two roles:

⌕Sub-agents — analysts

Each is spawned with the file key, node id, and the relevant outputs from earlier steps. It reads the design and returns findings and a fix-plan as structured text. It never writes.

✎Main session — the hands

Takes each plan and applies every Figma write it calls for, then verifies each write by an independent re-read. If a sub-agent can’t even read Figma, the orchestrator runs its spec inline instead.

Step 0

Platform confirmation — the only gate
⛔ PAUSE FOR USER CONFIRMATION

The orchestrator spawns the confirmation-agent, which parses the URL, reads every frame, builds the frame inventory, and recommends the target platform. The orchestrator presents this and waits for the user to confirm. That confirmation selects the one master design system the rest of the run obeys — not a loose colour library, not a UI kit, and any other subscribed library is flagged as out of place.

The pipeline

Spawn analyst → apply writes → verify

After the gate, the orchestrator works through nine build-and-fix sub-agents in strict sequence. For each: spawn the agent, receive the plan, apply the writes, verify by re-read.

1

File setup file-creator-agent

Rename the design page to “Ready for engineering”, create a separate “Design playground” page and a blue desktop thumbnail set as the file thumbnail, and import the master design system’s variable collections. (The file rename itself is flagged as a manual step.)

2

Frame naming file-renamer-agent

Rename every frame to the canonical convention — platform / area / feature / module / screen — state — ensuring each name is unique.

3

Sectioning section-splitter

Create one Figma section per sub-flow, reparent loose frames into them, top-align and pad every screen per the section recipe, and confirm no frame spills outside its section.

4

Token conformance token-matcher

Snap every numeric value to the nearest token within ±4px and bind colour, text, shadow and spacing variables; set the correct master-system mode; deliberately skip colours inside component instances.

5

Data correctness data-matcher

Replace placeholder and lorem text; correct dates, values, percentages, signs, sums and internal logic; flag unknown business rules rather than guessing.

6

Edge & missing states edge-finder

Clone the nearest existing frame into a “🆕 Proposed — <feature>” section and adapt it to the real missing state — never drawn from scratch. ⛔ Then the run pauses for the designer to review the proposed designs.

7

Motion guide animation-interaction-suggester

Render a dedicated “🎞️ Interaction & Animation Guide” frame placed above the first section.

8

Annotations annotater-agent

Write navigation, API, interaction and QA annotations onto every key frame, marking any unknown API call as TBD rather than inventing it.

9

Navigation map redirection-helper

Render a separate “🧭 Redirections & Navigation” frame mapping every touch-point to its target, flagging any dead-end call-to-action.

Quality gate

The reviewer — loop until 100% green

The orchestrator spawns the read-only reviewer. It audits the design screen by screen and frame by frame across every dimension at once — naming, sectioning and alignment, tokens and mode, data, maths and logic, edge states, annotations and redirections — and returns a per-frame pass/fail scorecard.

✓All green

The run proceeds to the flow chart.

↻Any failure

The reviewer names the exact owning agent and frames. The orchestrator re-spawns that agent on just those frames, re-applies the writes, and re-runs the reviewer. This loops until the scorecard is 100% green. The flow chart is forbidden from running before then.

Step 12

The flow chart — only after green

The orchestrator spawns the user-flow-creator, which returns a node-and-edge data model plus a build spec. The chart is then built in the main session through the Figma write channel (it cannot be drawn from a sub-agent). The result is a modern, neat, interaction-level user-flow chart with no crossing lines, API badges on relevant edges, and spacing bound to design-system variables. Self-check: no connector line over a tile, no tile overlapping another, crossings minimised, and every screen present.

Step 13

Self-check & report

Finally the orchestrator spawns the self-check-reporter (Figma-read + Slack). It runs the final verification pass — frame names, zero section spill and proper padding, master-system conformance, and a clean flow chart — and emits an audit report structured as found · fixed · needs human call. It then posts a bulletin to Slack #pre-iprd-test: verdict, counts, skip rate, and deep-links to the deliverables — owners named by role only, in plain text, never as images.

Always true

The standing rules
  • Sub-agents analyse; the main session writes — and verifies every write by reading it back (a read in the same call right after a create/move is stale, so it’s done separately).
  • The flow chart only runs after a fully green review.
  • No mid-run questions after Step 0, and nothing may be skipped — a skipped or partial step counts as a defect.
  • Frames are never deleted — they are hidden by setting them invisible; frames are reparented before any section is removed (removing a section deletes its children).
  • Endpoints and screens are never invented — unknowns are marked TBD and missing screens are cloned, never drawn.
  • One master design system per run, with only three permitted pauses: the platform confirm, a missing-master-system approval, and the edge-finder designer-review gate.
